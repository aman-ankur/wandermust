# Travel Optimizer — Design Spec

## Overview

A LangGraph-based multi-agent system that finds the optimal travel window for a destination given flexible dates. Balances weather, flight prices, and hotel costs using a hybrid approach: historical patterns to generate candidate windows, then live API checks on those candidates.

**Purpose:** Personal tool + learning project for multi-agent AI architecture.

**Stack:** Python, LangGraph, Streamlit, OpenRouter/Claude/any LLM, Amadeus API, Open-Meteo API.

---

## Architecture

### Agents (6 nodes in the graph)

| Agent | Type | Role |
|-------|------|------|
| Supervisor | Orchestrator | Parses input, generates candidate date windows, dispatches data agents |
| Weather Agent | Data fetcher | Queries Open-Meteo for temperature, rain, humidity per candidate window |
| Flight Agent | Data fetcher | Queries Amadeus Flight Offers for prices per candidate window |
| Hotel Agent | Data fetcher | Queries Amadeus Hotel Search for avg nightly rates per candidate window |
| Scorer | Deterministic | Normalizes all data 0-1, applies weighted scoring, ranks windows |
| Synthesizer | LLM-powered | Takes top 3-5 windows and generates human-readable recommendation |

### Flow

```
[User Input]
     |
[Supervisor] -> generates N candidate windows (e.g., weekly rolling blocks)
     |
[Weather] --\
[Flights] ---+-- parallel via LangGraph fan-out
[Hotels] ---/
     |
[Scorer] -> ranked list
     |
[Synthesizer] -> natural language recommendation
     |
[Streamlit UI] -> tables, charts, top picks
```

**LLM usage:** Only the Supervisor (input parsing) and Synthesizer (explanation) use an LLM. Data agents and scorer are deterministic.

## State & Data Model

```python
class TravelState(TypedDict):
    # Input
    destination: str              # "Tokyo, Japan"
    origin: str                   # "Bangalore, India" (default: BLR)
    date_range: tuple[str, str]   # ("2026-06-01", "2026-09-30")
    duration_days: int            # 7
    num_travelers: int            # 1
    budget_max: float | None      # optional ceiling
    priorities: dict              # {"weather": 0.4, "flights": 0.3, "hotels": 0.3}

    # Generated by Supervisor
    candidate_windows: list[dict] # [{"start": "2026-06-01", "end": "2026-06-07"}, ...]

    # Filled by data agents
    weather_data: list[dict]      # [{window, avg_temp, rain_days, humidity, score}]
    flight_data: list[dict]       # [{window, min_price, avg_price, currency, score}]
    hotel_data: list[dict]        # [{window, avg_nightly, currency, score}]

    # Filled by scorer
    ranked_windows: list[dict]    # [{window, weather_score, flight_score, hotel_score, total}]

    # Filled by synthesizer
    recommendation: str           # natural language summary
```

**Candidate window generation:** Supervisor creates rolling windows. For a 4-month range with 7-day trips, that's ~16 windows. Well within free API tier limits.

**Scoring:** Each data agent normalizes its own data to 0-1 (best = 1). Scorer applies user priority weights and sums.

## API Integration

### Open-Meteo (Weather)
- Endpoint: Historical Weather API (past dates) + Climate API (future dates, uses 30-year normals)
- Selection logic: if window end date < today, use Historical API; otherwise use Climate API
- Free, no API key needed, no rate limits
- Query: lat/lon (geocode destination first) + date range
- Returns: daily temp, precipitation, humidity
- Scoring: penalize rain days, extreme heat/cold. Ideal range configurable (e.g., 20-28C)

### Amadeus (Flights)
- Endpoint: `Flight Offers Search` — returns cheapest flights for origin/destination/date
- Free tier: 500 calls/month
- ~32 calls per search (16 windows x 2 for outbound + return)
- Need: Amadeus self-service API key (developers.amadeus.com)
- Uses IATA codes — city-to-IATA lookup via Amadeus Airport & City Search endpoint

### Amadeus (Hotels)
- Endpoint: `Hotel List` -> `Hotel Offers` — find hotels in city, then get prices
- Same free tier, shared 500 calls
- Strategy: cache hotel list (1 call), then 1 `Hotel Offers` call per window with `bestRateOnly=true` to get cheapest available rate across cached hotels. That's 1 + 16 = 17 calls total — NOT per-hotel queries.
- Fallback: if Hotel Offers returns empty for a window, mark that window as "no data" rather than retrying

### Geocoding
- Open-Meteo includes a free geocoding API — no extra dependency needed

### Rate Limit Strategy
- 16 windows: ~32 flight calls + 17 hotel calls + 16 weather calls (free) = ~65 Amadeus calls per search
- In-memory TTL cache (24h) so re-running with tweaked priorities doesn't re-fetch
- Can add SQLite cache later for persistence

### Historical Data Fallback
When live API calls fail or quota is exhausted, fall back to historical data:

**SQLite local database** stores every successful API response:
- `flight_prices(origin, destination, departure_date, price, currency, fetched_at)`
- `hotel_prices(city, checkin_date, avg_nightly, currency, fetched_at)`

**Fallback logic per agent:**
1. Try live API call
2. If it fails (rate limit, timeout, quota exhausted), check SQLite for same route + similar date window from prior searches
3. If historical data exists, use it but flag it in the UI as "estimated from historical data (fetched on {date})"
4. If no historical data either, mark that dimension as unavailable — scorer skips it and reweights remaining dimensions

**Over time** the database builds up a price history, making fallback data increasingly useful. This also enables future features like "price trend charts" showing how costs vary month-to-month for a route.

### Currency Handling
- Force INR via Amadeus `currencyCode` request parameter on all flight and hotel queries (configurable in .env)
- All scoring and display uses a single currency — no conversion needed

## Streamlit UI

### Single page, three sections:

**1. Input sidebar**
- Destination (text input with autocomplete)
- Origin city (text input, default: "Bangalore")
- Date range (date range picker)
- Trip duration (slider, 3-21 days)
- Number of travelers (number input)
- Budget ceiling (optional number input)
- Priority sliders (weather / flights / hotels — normalized internally, users set relative importance without worrying about summing to 1.0)
- "Find Best Time" button

**2. Results area**
- Top 3 recommendations as cards (date window, total score, estimated cost breakdown)
- LLM-generated summary explaining why these windows won
- Expandable details per recommendation

**3. Comparison charts**
- Line chart: flight price across all candidate windows
- Line chart: hotel avg nightly rate across windows
- Bar chart: weather score across windows
- Combined overlay showing the total weighted score

**Loading state:** Progress indicator showing which agents have completed.

**No auth, no login.** API keys live in `.env`.

## Error Handling

- If one data agent fails (e.g., Amadeus rate limited), others still complete. Scorer works with available data and flags the missing dimension.
- Amadeus token refresh handled automatically (tokens expire every 30 min).
- Network timeouts: 10s per API call, 3 retries with backoff.
- If destination can't be geocoded or has no IATA code, fail fast with clear UI message.
- LLM failures (Supervisor/Synthesizer): retry with backoff. If LLM is down, Supervisor falls back to simple string parsing, Synthesizer returns raw ranked data without natural language summary.

## Testing

- Unit tests per agent with fixture/mock API responses (no live keys needed)
- Scorer edge cases: missing data from a failed agent, all-equal scores, single candidate window
- Integration test: run full graph with recorded API responses and verify output shape
- Streamlit: manual testing (no UI test framework for MVP)

## Key Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Framework | LangGraph | Best for learning agent orchestration, model-agnostic, natural graph fit |
| Price strategy | Live API on candidate windows | Candidate windows keep call volume low; cache prevents redundant fetches |
| LLM provider | Any via OpenRouter / direct SDK | Learning flexibility, not locked to one provider |
| Data: flights + hotels | Amadeus free tier | Single signup, covers both, 500 calls/month |
| Data: weather | Open-Meteo | Free, no key, no rate limits |
| UI | Streamlit | Best for dashboards/charts, fast to build |
| Target user | Personal tool | CLI-level polish, no auth/onboarding overhead |
